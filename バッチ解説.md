# バッチ処理コード解説

## 全体の流れ

```
main.py
  ├─ EdinetClient.fetch_document_list_range()  # 日付範囲で書類一覧を取得
  ├─ EdinetClient.download_pdf()               # PDF取得
  ├─ PdfExtractor.extract()                    # テキスト抽出
  └─ EdinetIndexer.bulk_index()                # OpenSearchに投入
```

## main.py

### `build_doc(meta, body)`
EDINETのAPIレスポンス（meta）と抽出したテキスト（body）を組み合わせて、
OpenSearchに投入する1件分のdictを作る。

```python
def build_doc(meta: dict, body: str) -> dict:
    return {
        "doc_id":       meta.get("docID", ""),        # 書類ID（S100TS7Pのような形式）
        "company_name": meta.get("filerName", ""),    # 企業名
        "edinet_code":  meta.get("edinetCode", ""),   # EDINETコード（E01777など）
        "form_code":    meta.get("formCode", ""),     # 書類種別コード（030000など）
        "form_name":    meta.get("docDescription",""),# 書類名（有価証券報告書－第107期...）
        "period_start": meta.get("periodStart"),      # 対象期間の開始日
        "period_end":   meta.get("periodEnd"),        # 対象期間の終了日
        "submitted_at": meta.get("submitDateTime","")[:10],  # 提出日（日付部分だけ取り出す）
        "body":         body,                         # 抽出したテキスト本文
    }
```

### `run(start, end)`
バッチのメイン処理。

```python
batch = []
for i, meta in enumerate(docs_meta, 1):
    ...
    batch.append(build_doc(meta, body))

    if len(batch) >= 20:          # 20件溜まったら一括投入
        indexer.bulk_index(batch)
        batch.clear()

if batch:                         # 端数分も最後に投入
    indexer.bulk_index(batch)
```

**なぜ20件ずつ投入するか**
1件ずつ投入するより、まとめて送る（bulk）方がOpenSearchへのHTTPリクエスト数が減って速い。
20は特に意味はなく、メモリと速度のバランスで適当に決めた値。

## edinet/client.py

### `fetch_document_list(target_date)`
EDINET APIは**1日単位でしか書類一覧を返さない**設計になっている。
`type=2` を渡すと書類メタデータに加えて提出件数も含まれる（`type=1` はメタのみ）。

```python
resp = self._get(url, {"date": target_date.isoformat(), "type": "2"})
results = data.get("results", [])
return [
    r for r in results
    if r.get("formCode") in TARGET_FORM_CODES   # 有報・四半期・半期のみ
    and r.get("edinetCode") in self.edinet_codes # 対象企業のみ
]
```

### `fetch_document_list_range(start, end)`
上記を日付ループで繰り返して期間全体の書類を集める。

```python
time.sleep(interval_sec)  # デフォルト0.5秒
```
EDINETのAPIには明示的なレート制限の記載はないが、負荷をかけないために0.5秒待っている。
1年分スキャンすると365日 × 0.5秒 = 約3分かかる。

### `download_pdf(doc_id)`
`type=2` でPDFバイナリを取得。
EDINETのtype指定の意味：
- `type=1`: 提出書類本体（XBRL/HTML）
- `type=2`: PDF ← ここで使っている
- `type=4`: 添付書類

## edinet/extractor.py

### なぜ抽象クラスにしているか
```python
class DocumentExtractor(ABC):
    @abstractmethod
    def extract(self, raw: bytes) -> str:
        ...
```
将来XBRLに対応するとき、`XbrlExtractor(DocumentExtractor)` を追加して
`get_extractor("xbrl")` と呼ぶだけで差し替えられる設計にしている。
`main.py` 側は何も変えなくていい。

### サイズ・ページ制限
```python
MAX_PAGES = 30          # 先頭30ページまで
MAX_BYTES = 10 * 1024 * 1024  # 10MB超はスキップ
```
有価証券報告書は500ページを超えるものもある。
全ページ処理しようとすると1件に数分かかる。
重要な情報（事業概況・リスク・財務ハイライト）は前半に集中しているので、
30ページで検索品質は大きく落ちない。

### `_clean(text)`
```python
lines = [line.strip() for line in text.splitlines()]
lines = [line for line in lines if line]
return "\n".join(lines)
```
pdfminerで抽出したテキストは空行・余分なスペースが多い。
- 各行の前後のスペースを除去
- 空行を除去

これだけ。複雑な正規化はしていない。

## opensearch/indexer.py

### マッピング定義
```python
MAPPINGS = {
    "mappings": {
        "properties": {
            "company_name": {"type": "text", "analyzer": "kuromoji"},
            "body":         {"type": "text", "analyzer": "kuromoji"},
            "form_code":    {"type": "keyword"},  # keywordは完全一致用
            ...
        }
    },
    "settings": {
        "analysis": {
            "analyzer": {
                "kuromoji": {
                    "type":      "custom",
                    "tokenizer": "kuromoji_tokenizer",
                    "filter": [
                        "kuromoji_baseform",        # 動詞・形容詞を原形に正規化
                        "kuromoji_part_of_speech",  # 助詞・助動詞を除去
                        "cjk_width",                # 全角→半角正規化
                        "kuromoji_stemmer",         # 長音正規化
                        "lowercase",                # 英字小文字化
                    ]
                }
            }
        }
    }
}
```

**重要**: `"analyzer": "kuromoji"` はsettingsで定義したカスタムアナライザー。
`kuromoji_tokenizer` 単体ではなく、5つのフィルターを組み合わせることで助詞除去・正規化が実現される。

**`text` vs `keyword` の使い分け**
- `text`: 全文検索したいフィールド。kuromojiで形態素解析される。`body`・`company_name`
- `keyword`: 完全一致・フィルタ・集計に使うフィールド。`form_code`・`edinet_code`

**`company_name.keyword` について**
`company_name` → 形態素解析して全文検索
`company_name.keyword` → 完全一致・集計（`aggs`）用（明示的に定義が必要）

### `_ensure_index()`
```python
if not self.client.indices.exists(index=INDEX_NAME):
    self.client.indices.create(index=INDEX_NAME, body=MAPPINGS)
```
インデックスがなければ作る、あれば何もしない。
**注意**: マッピングを変えたいときはインデックスを削除してから再作成する必要がある。
既存インデックスのマッピングは後から変更できない（フィールド追加は可能）。

### `bulk_index(docs)`
```python
actions = [
    {"_index": INDEX_NAME, "_id": d["doc_id"], "_source": d}
    for d in docs
]
success, errors = helpers.bulk(self.client, actions, raise_on_error=False)
```
`_id` に `doc_id`（EDINETの書類ID）を使っているので、
同じ書類を再投入しても上書きされる（重複しない）。

## .env の内容

```
EDINET_API_KEY=...         # EDINETのAPIキー
OPENSEARCH_HOST=https://search-xxxxx.ap-northeast-1.es.amazonaws.com  # AWS OpenSearch Service エンドポイント
OPENSEARCH_INDEX=edinet-reports
```

※ AWS OpenSearch Service を使う場合は `OPENSEARCH_HOST` をAWSのエンドポイントURLに変更するだけ。
  IPベースのアクセスポリシーであれば認証コードの変更は不要。

## よくあるトラブル

| 症状 | 原因 | 対処 |
|---|---|---|
| `401 Access denied` | APIキーが間違い or スペース混入 | `.env` を確認 |
| `0件` しか取れない | 休日・formCode指定ミス | 日付・コードを確認 |
| `TypeError: unsupported operand` | Python 3.9で `set[str] \| None` 構文 | `Optional[set]` を使う |
| 処理が終わらない | 大きいPDFを全ページ処理している | `MAX_PAGES` を下げる |
| マッピングエラー | インデックスが古い設定のまま | インデックス削除 → 再作成 |
